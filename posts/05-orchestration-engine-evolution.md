# 从规则引擎到LLM动态编排——多Agent编排引擎的演进路线与渐进式升级策略

> 🎯 **目标读者**：已经在做多 Agent 编排、当前用规则引擎或静态 DAG，但开始感到"分支越来越多、规则越来越难维护"的后端架构师。这篇文章帮你理清编排引擎的演进方向，以及什么时候该升级、怎么渐进式升级。
>
> 本文是 [《AI Agent 多智能体编排架构实战》](/posts/04-agentforge-architecture-evolution) 的纵深补充。在那篇文章的 V3 架构中，我用 YAML 条件分支实现了"动态路由"，但诚实地说——**本质还是规则引擎**。这篇文章要回答的问题是：规则引擎之后，下一步该怎么走？

## 一、规则引擎不是终点，但它是一个好起点

在 [主文](/posts/04-agentforge-architecture-evolution) 的 V3 架构中，编排引擎通过 YAML 配置实现条件分支路由：

```yaml
steps:
  - agent: code-review
    on_complete:
      - condition: "result.severity == 'critical'"
        next: security-scan
      - condition: "default"
        next: test-execution
```

这套方案在 Agent 数量 < 5、场景相对固定时非常好用——确定性高、可审计、调试简单、成本可控。

**但它有三个天花板**：

1. **分支爆炸**：每增加一个 Agent 或一个场景，YAML 的条件分支数指数增长。当分支数超过 20，维护 YAML 的成本已经超过让 LLM 来做路由决策的推理成本。
2. **规则滞后**：业务场景变化快，规则变更频率从月级变成周级，人工更新规则跟不上。
3. **长尾场景无法覆盖**：总有一些请求无法被预定义的规则覆盖，只能 fallback 到人工处理。

这三个信号出现任何一个，就该考虑引入更"动态"的编排能力了。

2024-2026 年，多智能体编排领域已经形成了**三个清晰的技术方向**，以及一个**生产环境的共识方案**。

---

## 二、三大演进方向

### 方向一：Plan-and-Execute（规划-执行分离）

**核心思路**：把编排拆成两个阶段——Planner（强推理模型）生成执行计划，Executor（轻量模型或确定性代码）逐项执行。执行失败时触发 Re-Plan，而不是从头来过。

```
用户请求 → Planner（强模型，如 GPT-4/o3）
              ├─ 生成结构化步骤：[Step1, Step2, Step3]
              → Executor 逐项执行
              ├─ Step1 成功 → 执行 Step2
              ├─ Step2 失败 → 触发 Re-Planner 重新规划剩余步骤
              └─ 直到所有步骤完成或达到最大重试
```

LangGraph 的 Plan-and-Execute 模板是这个模式的典型实现。2025 年的安全架构研究进一步论证了这种模式的优势：将战略规划与战术执行分离，天然具有**控制流完整性**，比 ReAct 等反应式模式更能抵抗间接 prompt 注入攻击。

**优势**：
- 全局视野好——Planner 能看到完整任务，不会"走一步看一步"
- 可审计——计划可以被人审核和修改
- Token 成本可控——只在规划和 Re-Plan 时调用强模型

**局限**：
- 初始规划可能过时——执行中出现了新信息，Planner 无法提前预知
- 对探索性任务（调试、研究）效果差——Planner 缺乏足够上下文来制定有效计划
- Re-Plan 是"修正"而非"预判"

**什么时候用**：任务步骤可预测、需要全局视野的管线型任务——数据管线、多文件代码变更、合规工作流。

---

### 方向二：集中式动态编排器（CORAL / Puppeteer / DAAO）

**核心思路**：引入一个**专用的编排 Agent**，在运行时根据当前状态动态决定"下一步激活哪个 Agent"。不是按预定义的计划走，而是每一步都实时判断。

这是 2024-2026 年最活跃的研究方向，有几个重要的代表性工作：

**CORAL 信息流编排**：直接用信息流 orchestrator 持续监控任务进度，通过 A2A 通信工具动态协调 Agent，无需预定义工作流。在 GAIA 基准上达到 **63.64%**，比规则驱动的 OWL 方案高出 **8.49pp**（55.15%），且 token 消耗相当。CORAL 的核心论点是：工作流本质是规则决策树，有两个根本局限——需要大量人工枚举任务状态、无法穷尽复杂任务的状态空间。

**Puppeteer 模式**（2025）：集中式 orchestrator 在每个时间步根据当前状态动态选择激活哪个 Agent，通过 RL 优化编排策略，奖励函数显式平衡解质量和计算成本。关键发现是：演化出的拓扑呈现更紧凑、**循环的推理结构**——暗示循环推理（自我验证/修正）比纯线性流水线更有效。

**DAAO 难度感知编排**（The ACM Web Conference 2026）：根据查询难度自适应调整工作流深度、算子选择和 LLM 分配。包含 VAE 难度估计器，在 6 个基准上均有提升。这个思路非常务实——不是所有任务都需要动态编排，简单查询走规则就够了，只有复杂查询才需要 LLM 介入。

**优势**：
- 能处理长尾场景——不依赖预定义规则
- 自适应能力强——根据实时状态调整策略
- 学术基准表现优异

**局限**：
- 编排器本身成为单点故障和性能瓶颈
- 非确定性导致调试和审计困难
- 每一步都需要 LLM 调用，延迟和成本显著高于静态流程
- 需要精心的 Guardrail 和 Human-in-the-Loop 机制

**什么时候用**：开放域复杂推理、任务类型多样且不可预判、需要自适应成本-质量权衡的系统。

---

### 方向三：运行时 DAG 生成（AFlow / ADAS）

**核心思路**：不是让 LLM 在预定义的选项中选，而是让 LLM 或搜索算法在运行时**直接生成/优化工作流的 DAG 结构**。

**AFlow**（ICLR）：使用**蒙特卡罗树搜索（MCTS）**探索工作流空间。在 6 个基准上平均提升 5.7%，且使更小的模型以仅 **4.55% 的推理成本**在特定任务上超过 GPT-4o。这个思路的核心价值不是"让 LLM 更聪明"，而是"用更聪明的搜索算法让便宜模型也能干好活"。

**ADAS**（Automated Design of Agentic Systems）：在**代码空间**中搜索 Agent 设计——Meta Agent 迭代编程新 Agent。自动发现的 Agent 在编码、科学、数学等域上**显著超越手工设计 Agent**。

**GPTSwarm**（ICML 2024）：将 Agent 描述为可优化的计算图，通过 RL 优化边的连接概率，自动发现最优的 Agent 间通信拓扑。

**但这里有一个硬约束**：WorfBench 基准揭示，即使是 GPT-4，**在图规划与序列规划之间也存在约 15% 的差距**——LLM 生成正确的 DAG 结构仍然不可靠。生成线性的步骤序列相对容易，但生成正确的图结构（哪些步骤可以并行、哪些有依赖关系）要难得多。

**优势**：
- 能发现人类想不到的工作流结构
- 自动优化成本-质量权衡
- 理论上最"动态"的方案

**局限**：
- 搜索/优化过程本身消耗大量计算
- 发现的拓扑可解释性差
- LLM 生成正确的图结构仍不可靠（WorfBench 15% 差距）
- 目前主要在学术基准上验证，生产落地案例少

**什么时候用**：追求极致性价比、Agent 数量多且拓扑未知、愿意投入计算资源做工作流优化。

## 三、三大方向对比

| 维度 | Plan-and-Execute | 集中式动态编排器 | 运行时DAG生成 |
|------|-----------------|----------------|-------------|
| **核心机制** | LLM 规划 → 确定性执行 | LLM 每步实时决策 | LLM/搜索算法生成图结构 |
| **LLM 调用频次** | 低（规划 + 每步执行） | 高（每步 1 次决策） | 极高（搜索 + 优化） |
| **确定性** | 中（计划确定，执行确定） | 低（每步不确定） | 低（结构不确定） |
| **可审计性** | 高（计划可审核） | 中（决策链可追溯） | 低（搜索过程难解释） |
| **适用场景** | 管线型、步骤可预测 | 开放域、任务多样 | 拓扑未知、追求性价比 |
| **成熟度** | ⭐⭐⭐⭐⭐ 生产就绪 | ⭐⭐⭐⭐ 学术+早期生产 | ⭐⭐⭐ 学术前沿 |
| **代表方案** | LangGraph P&E | CORAL / Puppeteer / DAAO | AFlow / ADAS |

## 四、当前生产环境的共识：混合编排

说了这么多前沿方向，**实际生产中大家在怎么用？**

答案是：**不是"二选一"，而是"分层组合"。**

```
用户请求 → LLM Router（判断复杂度和类型）
              ├─ 简单/已知场景 → 规则引擎（YAML 路由，确定性，< 100ms）
              ├─ 中等复杂度 → Plan-and-Execute（LLM 规划 + 确定性执行）
              └─ 开放/长尾场景 → LLM 动态编排（全 LLM 驱动，需 Guardrail）
           → 结果聚合
```

**几个值得参考的生产案例：**

**Anthropic 多智能体研究系统**（Claude Research）：采用 Orchestrator-Worker 架构——Claude Opus 4 担任编排者负责规划与综合，多个 Claude Sonnet 4 作为子智能体并行搜索与提炼。核心设计原则是**上下文外化**：编排者将计划写入 memory 而非堆在上下文里，子智能体将发现写入共享文件系统而非通过聊天返回。在内部评测中，多智能体方案比单 Agent Claude Opus 4 **性能提升 90.2%**，代价是 token 消耗约为单 Agent 对话的 **15 倍**。Anthropic 同时明确给出了边界："需要所有 Agent 共享同一上下文、或 Agent 之间存在大量依赖的领域，目前不适合多智能体系统"。（来源：[Anthropic 官方博客](https://www.anthropic.com/engineering/claude-agent-teams)）

**Cognition/Devin 的生产原则**：**"多 Agent 可以贡献智能，但不要一起落笔"**——写代码和合并决策收束到单线程，其他 Agent 只贡献信息（搜索、审查、测试建议），不直接执行写入操作。实测数据：独立的编码 Agent + 审查 Agent（各自独立上下文），每个 PR 平均**多发现 2 个 bug，其中 58% 为严重级别**。Cognition 的解释是：代码中包含大量隐含决策（接口设计、状态流转、错误处理），多个 Agent 同时写代码会导致隐含决策冲突。这与 Anthropic 的结论互补——Anthropic 验证了"并行搜索"的价值，Cognition 划定了"并行写入"的边界。（来源：[Cognition 官方博文](https://cognition.ai/blog)、[CodeZine 报道](https://codezine.jp/news/detail/24057)）

**DAAO 难度路由**：用轻量 VAE 分类器判断查询难度，简单查询走规则/轻量模型，复杂查询才走 LLM 动态编排。这验证了一个直觉——**"知道什么时候不需要 LLM"比"什么都用 LLM"更重要**。

混合编排的核心设计原则：

| 原则 | 说明 |
|------|------|
| **确定性优先** | 硬约束用确定性代码检查，不信任 LLM 做合规判断 |
| **难度自适应** | 不是所有任务都需要 LLM 编排，简单查询走规则就够了 |
| **LLM 做概率性增值** | LLM 只在"评估""判断""规划"这些概率性任务上发挥价值 |
| **Guardrail 兜底** | LLM 编排的输出必须有边界检查，防止失控 |

## 五、什么时候该升级？

不是所有系统都需要 LLM 动态编排。以下信号出现时，才值得考虑：

| 信号 | 含义 | 建议动作 |
|------|------|---------|
| YAML 分支数 > 20 | 条件分支维护成本已超过 LLM 路由的推理成本 | 引入 LLM Router |
| 规则变更频率达周级 | 人工更新规则跟不上业务变化速度 | 引入 LLM Router 或 Plan-and-Execute |
| 长尾场景占比 > 30% | 超三成请求无法被现有规则覆盖 | 引入混合编排 |
| Agent 数量 > 5 且角色不稳定 | 静态拓扑无法应对动态变化的 Agent 能力组合 | 引入集中式动态编排器 |
| 规则维护成本 > 系统总成本 40% | 人力成本已超过 LLM 调用的增量成本 | 全面评估混合编排 |

**反面建议**：如果你的 Agent 数量 < 5、场景相对固定、规则变更频率是月级——**不要引入 LLM 编排**。规则引擎更便宜、更可靠、更好调试。引入 LLM 编排会增加延迟、成本和不确定性，收益微乎其微。

## 六、渐进式引入策略

如果确实需要升级，建议按以下路径逐步引入，**风险从低到高**：

### 第一步：引入 LLM Router（风险最低，收益最高）

```python
class LLMRouter:
    """用 LLM 替代 YAML 中的 if-else 路由决策"""
    
    async def route(self, task: str, agent_outputs: dict) -> str:
        """LLM 判断下一步应该调用哪个 Agent"""
        prompt = f"""
        当前任务：{task}
        已完成 Agent 的输出：{agent_outputs}
        可选的下一个 Agent：{self.available_agents}
        
        请根据当前状态，选择最合适的下一个 Agent。
        返回格式：{"agent_name": "...", "reason": "..."}
        """
        response = await self.llm.chat(prompt)
        return parse_response(response)
```

替代 YAML 中的条件分支，但**保留下游节点为确定性代码**。这是风险最低的第一步——LangGraph 的 Supervisor 模式就是为此设计的。

### 第二步：引入难度自适应

简单查询走规则/缓存，复杂查询走 LLM 规划。DAAO 的 VAE 难度估计器是可参考的实现。这一步的价值是**避免对简单任务过度编排**，节省 Token 成本。

### 第三步：引入 Re-Planner

执行失败时用 LLM 重新规划，替代硬编码的 fallback 路径。Plan-and-Execute 模式的 Re-Plan 循环是典型实现。这一步的价值是**让系统能自动修正错误**，而不是遇到意外就崩。

### 第四步：运行时拓扑生成（风险最高）

让 LLM 或搜索算法直接生成工作流 DAG 结构。**仅在封闭环境充分验证后再放开**。WorfBench 的 15% 图规划差距提示我们，LLM 生成正确的图结构还不够可靠。

## 七、写在最后

回到开头的问题：规则引擎之后，下一步该怎么走？

答案是：**不一定非要走。**

规则引擎在当前的 Agent 数量和场景复杂度下，可能是最优解。但你需要知道：
1. 行业前沿在哪（Plan-and-Execute / CORAL / AFlow 三个方向）
2. 什么时候该升级（5 个信号）
3. 怎么渐进式升级（4 步走，风险从低到高）
4. 生产环境的共识是什么（混合编排，确定性优先）

> 💡 **编排引擎的演进不是"一步到位"的架构升级，而是渐进式的能力叠加。** 关键是知道"什么时候该升级"，而不是"什么时候用最先进的"。最先进的方案不一定是最合适的——最合适的，是当前场景约束下的最优解。
